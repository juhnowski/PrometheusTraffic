# Запуск
Войдите в Nix-окружение проекта:
```bash
nix develop
```
Запустите стек Prometheus + Pushgateway:
```bash
start-prom
```
Запустите симуляцию нагрузки и экспорт файлов чанков:
```bash
run-bench
run-entropy
```
По завершении тестов остановите фоновые процессы:
```bash
stop-prom
```
# Специфика анализа файлов TSDB Prometheus
Файлы, собранные в директории ./prom_bench_blocks/, разделяются на два абсолютно разных типа данных:
- Файлы чанков (_chunk_000001.raw): Это чистый побайтовый поток, упакованный алгоритмом Gorilla. Алгоритм Gorilla сжимает временные ряды (float64) на уровне битовых дельт (XOR). Для алгоритмов дедупликации fixed-size или CDC это самый тяжелый стресс-тест. Данные Gorilla имеют очень высокую энтропию и сдвинуты на уровне бит. Если ваш алгоритм сможет найти дубликаты чанков между разными блоками, это покажет его превосходство над стандартным сжатием временных рядов.
- Файлы индексов (_index.raw): Содержат таблицы символов (String Table) со всеми строковыми именами метрик и лейблов (host, service, version). Они идут подряд в виде бинарных строк, выровненных по байтам. На файлах индексов ваш алгоритм дедупликации (особенно на сценарии 2 с денормализованными лейблами) должен показать очень высокий коэффициент сжатия.

# Описание
Стенд для Prometheus ставит перед алгоритмом постдедупликации уникальную задачу. Prometheus — это Time Series СУБД, но ее архитектура хранения данных существенно отличается от InfluxDB.
Начиная с версии 2.x, Prometheus использует архитектуру, вдохновленную LSM-деревьями. Данные группируются в блоки, каждый из которых охватывает жесткий временной интервал (по умолчанию — 2 часа).
Внутри папки такого двухчасового блока находятся:
- Файл chunks/000001 — здесь хранятся сырые бинарные метрики (временные метки и значения с плавающей запятой), сжатые алгоритмом Gorilla (XOR-сжатие).
- Файл index — инвертированный индекс для быстрого поиска метрик по именам и лейблам.
- Файлы метаданных meta.json и tombstones.

Так как Prometheus собирает данные методом Pull (сам опрашивает источники), для генерации трафика мы развернем в нашем Nix-окружении дополнительный легковесный компонент — Pushgateway. Питоновский скрипт будет отправлять туда метрики разных типов, Prometheus заберет их, а затем мы принудительно вызовем API-команду сжатия (snapshot), чтобы СУБД немедленно сформировала финальные блоки на диске.
